# 현대미술 경매 웹앱 — 제품 요구사항 문서 (PRD)

작성 목적: 초등 미술 감상·경매 수업용 웹앱 프로토타입을 만들기 위한 설계 합의 문서. 본 문서는 합의된 결정과 남은 열린 질문을 함께 담는다. 구현 단계의 1차 기준 문서로 사용한다.

상태: 프로토타입 운영 중
버전: **v1.2 (2026-06-30)**  |  앱 버전: **v0.2.0**
대상 스택: Cloudflare(이미지·호스팅) + Firebase(실시간 상태·데이터) + GitHub(버전관리) + Claude Design(시각 디자인)

## 1. 한 줄 요약

교사가 코드로 교실 세션을 열면, 학생들이 태블릿으로 미술관에 입장해 공통회랑·분기 작품을 감상(개인·수행평가 저장)하고, 모둠 단위로 실시간 경매에 참여한다. 경매가 끝나면 남은 돈과 낙찰작의 실제 감정가를 합산해 모둠 자산 순위를 발표하고, 작품의 실제 가치를 공개하며 "가치는 무엇으로 정해지는가"를 토론한다.


## 2. 교육적 목표 (수업 설계 근거)

핵심 질문은 활동지의 두 문항과 동일하다. (1) 예술가가 그린 것과 그렇지 않은 것의 차이, (2) 미술 작품의 가치 결정 기준.

감상 발문은 2022 개정 교육과정 성취기준 기반으로 학년군별로 다르게 제공한다.

- 3~4학년군: 호기심·직관적 느낌·자기 관점 존중. 관찰 유도형, 짧은 답·선택지 비계 병행.
- 5~6학년군: 분석적 사고·시대/지역 배경 이해·타인 관점 존중. 비교·분석·소통형.

경매의 학습 장치: 자산 = 남은 현금 + 낙찰작 감정가. 감정가보다 싸게 사면 자산이 늘고, 비싸게 사면 줄어든다. 즉 "좋아하는 것"과 "시장 가치"가 일치하지 않을 수 있음을 수치로 체험하게 만드는 것이 마무리 토론의 핵심이다.


## 3. 역할과 화면

세 종류의 화면이 존재한다. 사실상 한 앱 안의 세 가지 뷰(라우트)다.

| 화면 | 라우트 | 사용자 | 핵심 역할 |
|---|---|---|---|
| 교사 콘솔 | `/teacher` | 교사 1인 | 세션 생성·코드 발급, 작품 업로드, 단계 전환·잠금, 경매 진행(작품 넘기기/타이머 마감), 결과·감정가 공개, 데이터 내보내기 |
| 학생 태블릿 | `/play` | 학생(모둠 소속) | 입장, 감상 작성(개인 저장), 모둠 경매 입찰, 잔여 예산 확인, 결과 확인 |
| TV 송출 화면 | `/tv` | 교실 프로젝터(읽기 전용) | 현재 경매 작품, 현재 최고가·입찰 모둠, 누적 입찰 로그를 크게 표시 |

TV 송출 화면은 교사가 별도 URL을 전체화면으로 띄우는 공개 관전용 뷰다. 세 화면 모두 같은 세션(`/sessions/{code}`)을 구독하며, 교사의 단계 전환·타이머 마감이 진실의 원천(source of truth)이다.


## 4. 전체 흐름 (단계 모델)

학생 화면은 교사가 연 단계까지만 접근 가능하다. 단계 안에서는 학생 각자 속도로 진행하고, 단계 전환은 교사만 한다.

0. 프롤로그: 코드 입장 직후, 학년에 맞는 짧은 서사(동화)로 역할 동기를 부여. 3단계 정도로 넘기며 "건너뛰기" 버튼 제공.
1. 입장/로비: 코드 입력 → 번호·이름 입력 → 모둠 배정 확인 → 단계 대기.
2. 공통회랑: 공통 작품 감상(개인 저장, 수행평가 대상). 성취기준 발문에 답. 한 작품 감상을 마치면 그 작품의 해설을 보고 다음 작품으로 넘어간다. 공통회랑 작품 수(1~3)는 교사 설정.
3. 회랑 분기: 교사가 정한 개수의 문 중 선택 입장 → 문마다 다른 작품(교사 설정) 감상(개인 저장). 모둠원이 서로 다른 문을 골라 다양한 작품을 보게 유도. 분기 작품도 감상을 마치면 해설을 보고 넘어간다.
4. 경매장: 모둠별로 모여 회의 → 실시간 호가 경매. 공통회랑 + 분기 작품 전부가 경매 대상.
5. 결과 발표: 모둠 자산 순위 + 각 작품의 실제 감정가·출처 공개 → 교사 마무리 토론.

조기 종료 학생은 "다른 친구를 기다려요" 대기 화면을 본다. 이 화면에서 "내 감상 수정하기" 버튼으로 자신이 이미 작성한 감상을 다시 수정할 수 있다(교사가 다음 단계를 열기 전까지 = 현재 열린 단계 작품만 수정 가능, §10 확정).

> 디자인 참고: Claude Design 핸드오프의 흐름도는 프롤로그를 생략한 채 접속→로비→공통회랑으로 이어진다. 본 PRD는 프롤로그를 유지하되(§5.7), 핸드오프가 다루지 않은 화면(프롤로그·교사 콘솔·경매·TV)은 동일한 시각 토큰(§7.1)으로 새로 디자인한다.


## 5. 기능 상세

### 5.1 세션·입장

- 교사가 **반 이름(필수)·교사 이름(선택)** 입력 후 세션 생성 → 시스템이 6자리 입장 코드 자동 발급.
- 이 기기에서 접속했던 반은 **최근 반 목록**으로 저장(로컬). 반 이름·교사 이름·코드 표시, 클릭으로 재입장.
- 코드를 직접 입력해 들어가면 DB에서 반 정보를 읽어 목록에 자동 추가됨.
- 학생은 코드 + 번호 + 이름으로 입장. 계정·비밀번호 없음.
- 모둠 배정은 기본값이 **교사 사전 매핑**(번호→모둠). 교사가 옵션으로 "학생이 입장 시 모둠 선택"을 켤 수 있다.
- 동일 번호 중복 입장 처리(재접속 시 기존 세션 이어받기) 필요. 학생은 세션 내 **번호**를 키로 식별하며, 재접속 시 이름·기기(uid)를 갱신해 같은 레코드를 이어받는다.

교사 설정 요약(세션 생성 시 지정):

| 설정 | 내용 |
|---|---|
| 학년군 | 3~4 / 5~6 (발문 세트·프롤로그 분기) |
| 시작 자금 | 모든 모둠 동일 금액. 실제 감정가 규모를 고려해 교사가 지정 |
| 최소 호가 단위 | 한 번에 올릴 수 있는 최소 금액(입찰 증가폭) |
| 호가 타이머 길이 | 마감 카운트다운 시간(예: 10초) |
| 공통회랑 작품 수 | 1~3개 |
| 분기 문 개수 | 가변(교사 지정, 핸드오프 예시는 4문) |
| 분기별 작품 | 각 문에 배치할 작품(교사 업로드·지정) |
| 모둠 수 / 배정 방식 | 모둠 수, 사전 매핑 또는 학생 선택 |
| 감상 발문 | 학년군 기본 3종으로 시드 후 교사가 추가·삭제·수정 |
| 프롤로그 문구 | 학년군 기본 3단계로 시드 후 교사가 편집 |

### 5.2 작품 관리 (교사)

- 작품은 교사가 세션마다 직접 업로드.
- 업로드 시 필수 입력: 이미지, 작품명(임의 라벨 가능), **출처**, **감정가(실제 가격)**, **작품 해설**(감상 후 학생에게 보여줄 설명).
- 작품 속성 플래그: `경매 대상 여부`, `배치 위치`(공통회랑 / 분기N). 수행평가 대상은 공통회랑 작품으로 고정.
- 배치 규칙: 공통회랑 작품 수(1~3), 분기 문 개수, 각 문에 들어갈 작품을 교사가 지정.
- 경매 대상 작품 수가 모둠 수보다 적으면 업로드/세션 시작 단계에서 경고창을 띄운다(낙찰 못 받는 모둠 방지).
- 감정가·출처는 경매 중에는 숨김(결과 발표 단계에서 공개), 작품 해설은 감상 직후 학생에게 표시.
  - 프로토타입 구현 방식: 감정가·출처는 RTDB에 저장하되 **결과 단계 전에는 클라이언트에서 렌더하지 않는다**(완전 차단은 아님). 보안규칙으로 경로를 분리하는 강화는 향후 과제(§10 참고).
- 이미지 업로드는 Cloudflare에 저장하고 URL을 Firebase에 기록(아키텍처 §7 참고).
- **작품 비공개** (`🔒 공유 안 함`): 작품별로 설정 가능. 체크된 작품은 다른 반의 가져오기 목록에서 제외됨.
- **다른 반 작품 가져오기**: 모든 반의 공유 작품을 공통감상/선택감상/경매대상 탭으로 분류해 탐색. 원하는 작품을 체크 후 가져올 위치(공통감상실/선택감상실)를 지정해 복사. 반 선택 단계 없이 전체 공유 작품을 한번에 볼 수 있음.

### 5.3 감상 (개인·수행평가)

- 학년군 선택(교사가 세션 단위로 3~4 / 5~6 설정)에 따라 발문 기본 세트가 달라짐.
- 각 작품 감상에 **성취기준 발문을 기본 3종** 제공한다. 교사는 세션 설정에서 발문을 **추가·삭제·수정**할 수 있으며(학년군 기본 세트로 시드), 발문 개수는 가변이다. 데이터 모델에서 발문·답변은 고정 3개가 아니라 **배열**로 다룬다(§6).
- 감상 UI(핸드오프 갤러리 화면 기준): 우측에서 슬라이드되는 **감상 기록 패널**에 발문을 **한 번에 하나씩** 보여주고(저학년 글자 부담 완화), 자유 텍스트로 답을 적어 저장한다. 발문이 여러 개면 패널 안에서 스텝퍼로 다음 발문으로 넘어간다. 작품 간 이동은 좌우 화살표·진행 도트로 한다.
- 3~4학년 비계: 선택지 칸 + 문장 시작 도우미("나는 이 작품이 ___해서 ___하다") 병행. 음성 입력(Web Speech API) 옵션 검토.
- 한 작품의 감상을 제출하면 그 작품의 **해설 화면**이 나오고, 학생이 확인 후 다음 작품으로 넘어간다. (진행 도트로 이미 본 작품·감상은 재열람 가능.)
- 모든 답변은 학생 단위로 저장되어 교사가 열람·내보내기 가능.
- 학생은 교사가 다음 단계를 열기 전까지 자신의 감상을 다시 수정할 수 있다(현재 열린 단계 작품만, 대기 화면의 "내 감상 수정하기" 버튼 포함). 교사가 단계를 넘기면 이전 단계 감상은 읽기 전용이 된다.
- 수행평가 대상은 공통회랑 작품. 분기 작품도 개인 감상은 저장하되 평가 대상은 아니다(경매 회의용 자료로 활용).

발문 기본값(앱에 내장, 교사 편집 가능):

3~4학년군
- 이 작품을 구석구석 관찰해 볼까요? 화가에게 가장 먼저 물어보고 싶은 건 무엇인가요?
- 가장 눈에 띄거나 재미있는 부분은 어디인가요? 어떤 기분이 들었나요?
- 정답은 없어요. 제일 마음에 드는 점은? 내가 작가라면 제목을 뭐라고 지을까요?

5~6학년군
- 어느 시대·지역에서 만들어졌을까요? 당시 생활·사회 모습의 단서를 찾아볼까요?
- 작가가 전하려는 주제는? 그 주제를 위해 어떤 재료·표현·조형 요소를 썼나요?
- 친구는 같은 작품을 어떻게 다르게 봤나요? 왜 다르게 느꼈을지 존중하며 이야기해 볼까요?

### 5.4 경매 (모둠·실시간 호가)

- 입찰 주체는 **모둠 대표 1명**(대표 기기에서만 입찰). 나머지 모둠원은 회의로 참여하고 현재가를 함께 본다.
- 경매 진행 순서는 **랜덤**으로 작품이 등장(교사가 다음 작품으로 넘김).
- 입찰 방식: 대표가 "+호가 단위" 버튼으로 현재가에서 최소 호가 단위씩 올리는 방식을 기본으로(금액이 크므로 타이핑 부담 완화), 직접 금액 입력도 병행. 새 입찰은 현재 최고가보다 **최소 호가 단위 이상 높아야** 유효(동점·동시 입찰을 자연 해소). 경매 시작가는 0이며 첫 호가는 최소 호가 단위 이상.
- 예산 제약: 모둠의 입찰 가능액 = 시작 자금 − 이미 낙찰받은 작품의 **지불가** 합. 잔여 예산 초과 입찰은 차단/경고.
- 마감 타이머: 교사가 "타이머" 버튼을 누르면 마감 카운트다운이 시작된다. 카운트다운 동안 새 호가가 들어오면 입찰을 받아 카운트다운을 **자동으로 다시 시작(연장)**하고, 호가 없이 시간이 끝나면 현재 최고가 모둠에게 자동 낙찰. (경매사의 "마지막 호가" 흐름. §10 확정.)
- 유찰: 아무도 입찰하지 않은 채 마감되면 유찰 처리. 유찰된 작품에 한해 재경매 가능.
- TV 송출 화면과 모든 학생 화면에 현재 최고가·입찰 모둠·남은 타이머·누적 로그가 실시간 반영.
- 모든(또는 교사가 선택한) 작품 경매가 끝나면 경매 종료.

동시성: Firebase **트랜잭션**으로 "현재 최고가 갱신"을 원자적으로 처리해 두 모둠이 거의 동시에 같은 금액을 올려도 한쪽만 최고가가 되도록 한다.

### 5.5 결과·자산 순위

- 모둠 자산 = 남은 현금 + 낙찰받은 작품들의 **감정가** 합. (남은 현금 = 시작 자금 − 낙찰 지불가 합.)
- 자산 기준으로 모둠 순위 표시.
- 각 작품의 실제 감정가·출처 공개 화면 제공(교사가 설명).
- 모둠별 "지불가 vs 감정가" 비교를 시각적으로 보여주면 토론에 유용(이득/손해가 한눈에).

### 5.6 데이터 내보내기

- 학생별 감상 기록(작품·발문·답변)을 표(CSV)와 PDF로 내보내기.
- 모둠별 경매 결과(낙찰작·가격·잔액·자산·순위)를 표(CSV)와 PDF로 내보내기.
- 수행평가 기록용이므로 학생 이름·번호와 답변이 정렬되어 출력되어야 함.

### 5.8 슈퍼어드민 (`/admin`)

- 교사 암호와 별도의 관리자 암호로 보호 (최초 접속 시 설정, 이후 고정).
- 전체 세션을 실시간으로 모니터링: 반 이름·교사 이름·현재 단계·접속 학생 수·작품 수·감상 답안 수.
- 코드 또는 반 이름으로 검색, 최근 생성 순 정렬.
- 각 세션에서 교사 콘솔 바로 열기, TV 송출, 학생 입장 QR 링크 제공.
- 세션 데이터는 Firebase RTDB에 영구 저장 (`sessions/{code}/`).

### 5.7 프롤로그 (입장 서사)

- 코드 입장 직후 학년에 맞는 짧은 동화 형식의 서사로 역할 동기를 부여.
- 3단계(화면) 정도로 넘기며, 각 화면에 "건너뛰기" 버튼 제공(언제든 본 활동으로 진입).
- 학년군에 따라 문장 난이도·길이를 다르게(3~4학년은 더 짧고 쉬운 문장).
- 프롤로그 문구는 학년군 기본 3단계로 시드되며 **교사가 편집할 수 있다**(§10 확정). 세션에 저장.

내장 기본값(3단계)

3~4학년군
1. "오늘 우리는 특별한 미술관에 초대받았어요. 세상에 하나뿐인 그림들이 기다리고 있대요."
2. "그림들을 천천히 둘러보고, 마음에 드는 작품을 찾아보아요. 그리고 친구들과 이야기도 나눠요."
3. "마지막엔 경매장이 열려요! 우리 모둠이 가진 돈으로 갖고 싶은 그림을 골라 값을 불러요. 자, 출발!"

5~6학년군
1. "여러분은 오늘 한 미술관의 감정단으로 초대되었습니다. 작품의 가치를 직접 살펴보는 일이 맡겨졌어요."
2. "공통 전시실과 여러 전시실을 돌며 작품을 관찰하고, 무엇이 이 작품을 특별하게 만드는지 기록합니다."
3. "마지막은 경매. 모둠의 한정된 예산으로 어떤 작품에 얼마를 걸지 전략을 세워야 합니다. 가치를 보는 눈, 지금부터 시작합니다."


## 6. 데이터 모델

Firebase Realtime Database 기준. 세션 코드를 최상위 키로 사용하고 세 화면 모두 `/sessions/{code}` 하위를 구독한다.

```
/sessions/{code}
  /meta                          → SessionMeta      (세션 설정)
  /state                         → SessionState     ★ 단계 = 진실의 원천
  /content                       → SessionContent   (교사 편집: 프롤로그·발문)
  /groups/{groupId}              → Group
  /students/{number}             → Student          (번호를 키로 → 재입장 idempotent)
  /artworks/{artworkId}          → Artwork          (감정가·출처 포함, 결과 전 비표시)
  /appreciations/{number}/{artworkId} → Appreciation
  /auction/items/{artworkId}     → AuctionItem
  /auction/bids/{bidId}          → Bid              (push 키, 누적 로그)
  /results/{groupId}             → GroupResult
```

주요 필드(타입 개요)

| 엔티티 | 주요 필드 |
|---|---|
| SessionMeta | code, teacherUid, **className**, **teacherName**, gradeBand(3-4/5-6), startingFunds(원), minIncrement(원), timerSeconds, commonGalleryCount(1~3), branchDoorCount, groupAssignMode(preset/studentChoice), groupCount, createdAt |
| SessionState | phase(lobby/prologue/gallery/branch/auction/result), currentAuctionArtworkId? |
| SessionContent | prologue: string[], prompts: string[] (학년군 기본으로 시드, 교사 편집·가변) |
| Group | id, name, repStudentNumber?, remainingBudget(원), wonItems: { artworkId → 지불가 } |
| Student | number, name, groupId, isRep, uid, joinedAt, lastSeenAt |
| Artwork | id, imageUrl, title, source, appraisedValue(원), commentary, placement(common/branch:N), forAuction, order, **isPrivate**(공유 안 함 플래그) |
| Appreciation | studentNumber, artworkId, answers: string[] (발문 개수에 대응), submittedAt, updatedAt |
| AuctionItem | artworkId, status(pending/live/sold/passed), currentPrice(원), highBidGroupId?, timerState(idle/running), timerEndsAt?, order(랜덤) |
| Bid | id, artworkId, groupId, amount(원), at |
| GroupResult | groupId, remainingCash, wonAppraisedSum, asset(=remainingCash+wonAppraisedSum), rank |

확정 규칙
- 돈은 모두 **원 단위 정수**.
- 단계 접근 게이트는 `state.phase` 단일 값(학생은 phase까지만 접근).
- 타이머는 **자동 연장**(호가 들어오면 카운트다운 재시작).
- 감상 발문·답변은 **가변 배열**(고정 3개 아님).


## 7. 기술 아키텍처

- 호스팅: 정적 웹앱을 **Cloudflare Pages**로 배포. GitHub를 소스 저장소로 사용.
- 프론트엔드: React + Vite + TypeScript, Tailwind CSS.
- 실시간 상태·데이터: **Firebase Realtime Database**(저지연 입찰 때문에 Firestore 대신 RTDB). 단계 전환, 입찰, 감상 저장 모두 여기로 동기화. 새로고침·재접속 시 서버 상태로 복구.
- 인증: 학생은 **Firebase 익명 인증** + 코드 기반 입장, Firebase 보안 규칙으로 세션 격리. 교사도 프로토타입에서는 익명 인증으로 두되 세션 생성자(teacherUid)로 식별.
- 이미지: Cloudflare에 업로드(Cloudflare Images 또는 R2). 클라이언트에 API 키를 노출하지 않기 위해 **Cloudflare Worker**로 업로드를 중계 → 반환된 이미지 URL을 Firebase에 저장. API 키 등 시크릿은 Worker 환경변수로만 둔다.
- 설정·시크릿: 코드에 넣지 않고 `.env`로 주입. `.env.example`을 제공하고 README에 로컬 실행·배포법을 정리한다.

데이터 흐름 요약: 교사 콘솔/학생 태블릿/TV 화면이 모두 Firebase의 같은 세션을 구독한다. 교사의 단계 전환과 경매 타이머 마감이 진실의 원천이고, 나머지 화면은 이를 실시간 반영한다.

### 7.1 시각 정체성·디자인 토큰 (Claude Design 핸드오프 반영)

색

타이포

컴포넌트·모션 패턴

## 8. 비기능 요구사항

- 시간 설계: 1차시(약 40분) — 감상 20분, 경매 20분 목표. 단, 학급 규모·작품 수에 따라 초과 가능성 있으므로 세션 저장·이어하기를 기본 지원.
- 저학년 UX: 큰 버튼, 적은 글자, 현재 단계 시각 표시, 음성 입력·선택지 비계, 한 번에 하나의 발문.
- 안정성: 와이파이 끊김 후 재접속 시 상태 복구. 입찰은 트랜잭션으로 정합성 보장.
- 개인정보: 이름·번호 등 PII 최소화. 데이터는 세션에 귀속되며 교사가 열람·내보내기·삭제 가능. 가능하면 식별을 번호 중심으로 두고 이름은 표시·정렬용으로 한정.
- 접근성: 태블릿 가로/세로 대응, 글자 크기 확대 여지. 다크 배경에서 충분한 명도 대비 확보.
